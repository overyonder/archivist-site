How long does your game launcher take to load?

We hydrate 222,242 library entries in 420 ms.

Archivist is a highly visual way to explore the canon of video games. That means performance is not an implementation detail: it is part of how the collection feels. Search, timelines, verification and browsing need to stay immediate even when the library is absurdly large.

We are high-performance game developers at heart. We believe software needs to be fast to feel good. Also, if we are honest, we simply could not help ourselves.

The headline is deliberately precise: 210,840 files plus 11,402 directories—not 222,242 distinct games. The measured 420 ms begins when the already-running frontend requests its snapshot and ends when that snapshot is applied. A hot process reached the same applied state 1.167 seconds after launch.

01 / the constraint

Fast enough to disappear.

Archivist is closer to a media presentation system than a spreadsheet. A user can move from a platform timeline to a genre genealogy, inspect release evidence, browse cover art and launch a game. Any pause breaks the illusion that the entire history is simply there.

Our success criterion is 144 frames per second on every screen. That is a 6.94 ms frame budget. Startup work may use the brief interval in which the operating system is already creating the process, but no later interaction gets a licence to hitch.

Keep the complete working set resident. Keep its layout predictable. Move cold work beside the disks. Never make browsing wait for discovery.

This is why Archivist does not treat just-in-time loading as the primary architecture. The full selected catalogue stays native and resident. The storage snapshot is loaded up front. Rendering may be virtualised to visible rows, but the data itself is not evicted merely because it is off-screen.

02 / architecture

Compute where the bytes already live.

The first large gain was architectural. Pulling every ROM across NFS to a desktop process is wasteful before an algorithm even begins. The NAS Historian performs metadata discovery, logical archive hashing and manifest matching beside the storage pool, then sends a compact immutable projection to the desktop.

Appliance deployment

One static musl binary

No glibc, Electron or .NET runtime in the storage path. The statically linked x86-64 executable is designed to copy onto Linux NAS appliances such as TrueNAS SCALE.

Protocol split

HTTP bulk, WebSocket propagation

Bulk snapshots use one authenticated compressed response. A persistent Rust-to-Rust WebSocket carries status and generation notices without exposing historian credentials to JavaScript.

Mutable evidence

SQLite WAL, not a UI database

Per-file logical digests are restartable in a WAL store. Hashing happens outside short batched transactions; immutable UI snapshots are cached separately.

Safe semantics

Snapshot time is explicit

The UI says when storage was last scanned. A cached scan is useful evidence, not a false promise that an offline filesystem remained unchanged.

04 / resident catalogue

Two million rows before you can blink.

The catalogue compiler does not build a general-purpose relational database. It extracts the evidence Archivist actually needs, normalises binary digest keys and produces a strongest-digest lookup optimised for the validation join.

05 / format tournament

MessagePack survived—just not everywhere.

MessagePack remains a good fit for the small irregular header. It lost badly as the representation for hundreds of thousands of repetitive rows. The winning format is a hybrid: MessagePack metadata followed by aligned struct-of-arrays columns and a single NUL-separated basename blob.

What about tokenising “Pokémon”?

It was worth trying. In the path corpus, 69.4% of component occurrences fit within the top 128 tokens. But the repeated directory vocabulary was already captured by the parent index, while leaf filenames remained mostly unique. A second dictionary reduced bytes but added pointer chasing and reconstruction work.

06 / disks, digests and joins

Read once. Prove three things.

ROM manifests commonly provide CRC32, MD5 and SHA-1. Historian updates all three digest states during the same sequential read, including logical members inside ZIP containers. It does not read the file three times.

08 / shipped optimisation ledger

What actually made the binary.

  • ShippedNAS-side inventory, logical archive hashing, matching, filesystem notifications and guarded same-filesystem rename.
  • ShippedCRC32, MD5 and SHA-1 updated together during one read through reusable 1 MiB per-worker buffers.
  • ShippedAssembly-enabled digest crates with runtime CPU-feature dispatch; binary digest arrays instead of hexadecimal strings.
  • ShippedFixed-width FxHashMap evidence joins; the detailed two-million-row digest graph never crosses into JavaScript.
  • ShippedStatic policy pruning: 1,221 sources become 292 contributing sources, 313,914 retained releases and 1,239,634 unique strongest digests.
  • ShippedVersioned 50 MiB native catalogue artefact, memory-mapped on warm startup; policy invalidation ignores unrelated transport and dependency changes.
  • ShippedSQLite WAL with NORMAL synchronous mode, 256-record batches, passive checkpoints and hashing outside write transactions.
  • ShippedImmutable encoded snapshot cache, O(1) status counters and WebSocket generation propagation.
  • ShippedHybrid MessagePack + aligned SoA snapshot, gzip via zlib-rs, parent indices, relative paths and a single basename string blob.
  • ShippedDirect browser DecompressionStream, typed-array views and one UTF-8 decode; Rust replica decoding runs concurrently.
  • ShippedStartup request cleanup: the frontend consumes its resident snapshot and no longer greedily asks for manifests, intake listings or verification details.
  • ShippedEnd-to-end timestamp logging from process creation through HTTP, Tauri IPC, decode and frontend application.
09 / hardware acceleration audit

The fastest instructions are currently hidden by the VM.

The physical Intel N100 exposes AVX2, PCLMULQDQ, SHA-NI, VAES and VPCLMULQDQ. The TrueNAS VM is configured as x86-64-v2-AES, exposing SSE4.2 and AES but hiding PCLMUL, AVX2 and SHA-NI. Our portable static binary already contains runtime-dispatched assembly implementations, but a guest cannot select instructions its virtual CPU does not advertise.

This isolates a likely opportunity, not a final production claim. Host and guest are different execution environments and the run count is one. The 3.8× SHA-1 result is consistent with SHA-NI becoming visible; the combined path is the more relevant number. We should switch VM CPU type to host passthrough during a maintenance window, then rerun the full controlled benchmark.

A GPU is not automatically a faster hash engine. Files must still travel from storage into host memory and then across the PCIe/device boundary, while our production NAS is currently limited by spinning disks and file scheduling. GPU hashing becomes interesting only for large contiguous NVMe workloads or when a GPU is already processing the same bytes.

10 / what remains

We stopped where evidence stopped.

Not every idea became code. Some lost their benchmark; others need platform-specific work large enough to deserve their own milestone.

  • NextExpose the N100 host ISA to the TrueNAS VM and rerun hashing, historian startup and power measurements.
  • CandidateZFS snapshot-delta inventory, so a new scan can identify changed paths in one filesystem-native sweep.
  • CandidateLinux statx identity reuse for hard links; fs-verity digests where the filesystem can prove them; optional digest xattrs for portable rehash avoidance.
  • Candidategetdents64/statx scanner and io_uring reader providers, benchmarked separately on HDD, SATA SSD and NVMe rather than assumed faster.
  • CandidatePatch snapshots over WebSocket for small live changes. Startup should retain the single dense snapshot; a one-file mutation should not require another full projection.
  • CandidateSame-pool reflinks and copy_file_range for moves that cannot use rename(2).
  • RejectedWholesale JSON, CBOR, protobuf rows, FlatBuffer-style rows, lexical path tokens and a global filename dictionary. Each lost on total bytes, decode time or complexity.
  • RejectedA B-tree digest index. Ordered traversal is not the operation; exact digest lookup is.
  • DeferredPortable zero-copy shared memory into the WebView. Tauri does not provide a cross-platform borrowed Rust buffer to JavaScript, and custom platform bridges would still leave JavaScript object materialisation.
11 / the next ceiling

Native retained-mode Rust—without losing the look.

The current WebView boundary is no longer the catastrophic cost it was: the live snapshot applies in 420 ms and all heavy evidence remains native. But 144 Hz on every screen means controlling allocation, layout, text shaping, upload and draw submission every frame. A native renderer is the clearest route to that control.

We would not begin by choosing a generic widget toolkit and rebuilding every screen. The aesthetic is part of Archivist. The migration should treat the existing interface as a visual contract.

  1. Freeze the visual contract.Extract colours, type scale, spacing, borders, shadows, animation curves and responsive rules into shared design tokens. Capture reference screenshots for every state and viewport.
  2. Separate presentation models from DOM models.Move sorting, grouping, selection and row projection into Rust read models backed by the existing resident SoA catalogue. Keep full data resident; virtualise only visual nodes.
  3. Build one native performance screen.Use the largest timeline or verification tree as the vertical slice. Run it beside the current screen behind a feature flag and compare frame time, memory, input latency and pixel output.
  4. Render with winit + wgpu.Use persistent GPU buffers, instanced rectangles and images, a glyph atlas with cosmic-text/swash, explicit clipping and no per-frame heap allocation in steady state.
  5. Retain and invalidate.Keep a scene graph and layout cache. Recompute only dirty subtrees; update only changed GPU ranges; render only visible rows while every catalogue record remains immediately addressable.
  6. Preserve desktop quality.Add AccessKit accessibility, IME and clipboard support, keyboard/gamepad focus, high-DPI scaling and platform window semantics before replacing the corresponding WebView screen.
  7. Gate on measurable parity.Require perceptual screenshot parity, 144 Hz frame pacing on target hardware, bounded allocations and equal-or-better accessibility. Migrate screen by screen; keep the proven web implementation until each replacement wins.

Why not egui? It is excellent, but its immediate-mode ergonomics are not the same thing as the explicitly retained, dirty-region-controlled renderer we want. Why not Slint or Iced? Both are credible prototypes, and Slint in particular preserves declarative layout, but a custom wgpu presentation layer gives us the most direct control over this unusually data-heavy, game-like interface. We should prove that choice with the vertical slice before committing the whole product.

12 / methodology and honesty

Numbers need boundaries.

All timings on this page are development measurements from the copied corpus and hardware described beside them. Release binaries were used for comparative runs. Results are not blended across stages to invent a “two million games loaded” claim.

Benchmark mirror

12,297 DAT files · 1.77 GB copied locally

The selected policy contains 1,221 manifests. The exact OxyROMon/Igir compatible intersection contains 503 files and 487,201,103 bytes.

Live storage corpus

222,242 filesystem entries

210,840 files and 11,402 directories in the measured historian snapshot. These entries are not all distinct playable games.

Output semantics

Compiler ≠ database import

Archivist builds the resident strongest-digest evidence it needs. Competitors may perform additional relational, reporting or repair work.

Cache effects

Called out, not averaged away

ZFS ARC, Linux page cache and run order materially affect disk tests. Cold, warm and one-run figures are labelled where known.

The benchmark tools live with Archivist so future changes can be measured using the same inventory, combined-hash, matching and catalogue workloads. If a later implementation is slower, the page should change rather than the methodology.
