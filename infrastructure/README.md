# Archivist infrastructure

OpenTofu owns the hosted infrastructure around Archivist early access:

- Cloudflare Turnstile and SES-related DNS records;
- Amazon SES identity, MAIL FROM, configuration set, contact list and events;
- Amazon SNS delivery-event transport;
- the full-scope IAM administration identity used by OpenTofu;
- the least-privilege IAM runtime identity used by Supabase;
- the monthly AWS cost budget;
- the Supabase project settings, Edge Functions and function secrets.

The working `archivist` test project is in `ap-southeast-2`. OpenTofu creates
its `archivist-us` production replacement in `us-east-1`; it must not import
the Sydney project into that resource. Settings, functions, secrets and the SNS
webhook all derive the new project reference from
`supabase_project.archivist.id`.

The database schema remains in `../supabase/migrations`. Apply migrations to the
new US project before deploying function versions that depend on them. Preserve
the Sydney project until the US deployment passes the end-to-end test and its
consent history has been copied or archived.

`terraform_data.edge_functions` owns Edge Function deployment. Its replacement
trigger hashes the complete `supabase/functions` tree, and its apply invokes
Supabase's official CLI bundler from the toolchain pinned by `flake.lock`. The
provider's native Edge Function resource uploads entrypoints without preserving
the `../_shared` module layout, so it cannot deploy this source tree correctly.

## Credentials and state

OpenTofu state and saved plans are AES-GCM encrypted. Supply the encryption
passphrase as `TF_VAR_state_passphrase`; never put it in a `.tfvars` file.
Provider credentials use their standard environment variables:

- `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` for a provisioning identity;
- `CLOUDFLARE_API_TOKEN` with Turnstile and DNS read/write access;
- `SUPABASE_ACCESS_TOKEN` for the Management API.

The Supabase Management API token and the full-scope AWS provisioning access-key
pair are stored in SOPS. The provisioning user has no console password and is
deliberately retained with `AdministratorAccess`, matching the policy of keeping
encrypted full-administration credentials for managed services. It is separate
from the least-privilege SES runtime identity exposed to Supabase. After the
first AWS apply creates or adopts that runtime identity, its access key ID and
secret access key must also be added to SOPS and deployed to Supabase through
OpenTofu.

During the Sydney-to-US cutover, `aws_iam_access_key.edge_legacy` retains the
existing Sydney runtime key with destruction protection. AWS cannot reveal an
imported key's secret, so `aws_iam_access_key.edge` creates the second key used
by the US project. Remove the legacy key only after the US signup flow passes
end to end and the Sydney endpoint has been retired.

The database password is supplied as `TF_VAR_supabase_database_password`.
Operational secrets generated by OpenTofu are stored only in encrypted state
and copied into Supabase's secret store. Provider-managed `SUPABASE_*` values
remain owned by Supabase and are deliberately excluded.

The local state file is ignored by Git even though it is encrypted. Back it up
with the rest of the workstation's protected configuration data. Losing both
the state and its passphrase requires importing the live resources again.

## First adoption of existing resources

1. Copy `imports.tf.example` to `imports.auto.tf`.
2. Replace the AWS account ID and any AWS names that differ from the live
   resources. Do not add an import for the Sydney Supabase project.
3. Run `tofu init` and `tofu plan`.
4. Reconcile configuration until the plan imports resources without replacing
   them.
5. Apply once, then remove `imports.auto.tf`.

Never apply an import plan containing an unexpected destroy or replacement.
